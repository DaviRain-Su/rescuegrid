## Wallet evidence artifact — ready for manual click-through

`npm run wallet:evidence -- --format markdown --out .rescuegrid/wallet-clickthrough-evidence.md` prepares a gitignored artifact for the real Slush / standard Sui wallet create/revoke run. It is read-only: it snapshots frontend reachability/login guardrails plus public Worker root/runtime/readiness/chain-data posture, signer capability posture and external signer posture, and lists the exact fields that must be filled during browser QA, including owner address, create tx digest, `wrapper_id`, `mandate_id`, strategy hash, revoke tx digest, `Actual click-through completed: true`, and screenshots/evidence references for sign-in, wallet approvals, active policy, activity rows and revoked state. `npm run wallet:evidence:preflight` requires the local frontend, no-auto-connect sign-in boundary, explicit Worker read-only entry and Worker public status/readiness to be ready before manual QA. After the artifact is filled, `npm run wallet:evidence:verify -- --input .rescuegrid/wallet-clickthrough-evidence.md` rejects obvious secret assignments, requires the manual fields, then checks the create/revoke digests against Sui `PolicyCreated` / `PolicyRevoked` events and Worker detail create/revoke activity plus revoked-state reads; final `mission:readiness` requires that Worker detail evidence. The generated blank artifact or preflight pass is not proof that the extension click-through happened; it is the evidence template and environment check to run before the real browser run.

## Known gaps / next

1. **DBUSDC funding** — the only true remaining gap, and **self-funding is confirmed impossible** on this testnet: DBUSDC `mint` is TreasuryCap-gated (cap not public), DEEP `mint` returns `FunctionNotFound` on the current package, and a SUI→DBUSDC swap needs DEEP for taker fees (a zero-DEEP swap fills 0 even with a live bid). Needs an **external DBUSDC source** (DeepBook-team faucet, or an address that already holds DBUSDC/DEEP). Run `npm run funding:request -- --format markdown --out .rescuegrid/funding-request.md` for the current public BalanceManager id, coin types and missing amounts to send to the funding provider. Once the agent BalanceManager holds DBUSDC/DEEP: flip `EXECUTION_ENABLED=true`, run `npm run funding:watch -- --json` and `npm run funding:watch:report` to verify readiness without creating a policy, then use `npm run funding:watch -- --wait --run-demo --worker-url http://localhost:8787` or `npm run demo:execute`; strict execution will replay the execution PTB through `worker/src/executor-adapters.js` and fail unless the chain emits `AgentTradeExecuted` with spend increase.
2. **Browser wallet click-through** — connect Slush (testnet) and run create/revoke from the UI against `VITE_WORKER_URL`. The frontend wallet orchestration is mock-signer tested, including `PolicyCreated.wrapper_id` extraction and runtime activation; `npm run wallet:evidence:preflight` now checks local frontend/login/Worker readiness before manual QA; `npm run wallet:evidence -- --format markdown --out .rescuegrid/wallet-clickthrough-evidence.md` prepares the gitignored evidence artifact; `npm run wallet:evidence:verify -- --input .rescuegrid/wallet-clickthrough-evidence.md` validates explicit click-through completion, manual screenshots/evidence fields, Sui events and Worker detail reads. A real extension click-through with filled tx digests, wrapper/mandate ids and screenshots is still the final browser evidence.
3. **zkLogin live test** (optional) — the Enoki Google option is surfaced and helper-tested when configured; real OAuth click-through still needs browser evidence if using Enoki instead of a standard wallet.
